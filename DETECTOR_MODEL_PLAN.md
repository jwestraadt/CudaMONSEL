# Detector Geometric / Energy-Angle Filtering — Design Plan

Status: design only (no code yet).

## Motivation

The current `composite_image` simulation uses an idealized, omnidirectional
detector: any electron that escapes to vacuum is tallied, split only by a 50 eV
SE/BSE energy threshold and an SE1/SE2 *generation-direction* label. Real SEM
detectors collect a specific **energy and angular** sub-population, and different
detectors see different physics:

- An **annular / in-column BSE** detector collects high-energy backscatter over a
  ring of take-off angles — it sees **bulk mean-Z** contrast.
- An **in-lens / in-column SE** detector (e.g. Trinity T3) collects low-energy
  true SE1 funnelled up the column — it sees **surface SE emissivity**.
- A **chamber Everhart–Thornley (ETD)** SE detector is dominated by bias-collected
  SE plus SE3 generated by BSE on the pole-piece/chamber — it tends to track BSE.

This explains experimental observations that the single blended yield cannot:
a dense refractory phase (sigma, W-rich M23C6) reads **bright in BSE** yet **dark
on an in-lens SE detector** at the same time. The model needs configurable
detector channels to reproduce this.

## Key enabler: the escape state is already computed

No new trajectory tracking is required. At the escape event both backends already
have the full kinematics; they simply are not used for filtering yet:

- GPU: `recordEscape` (CompositeImage_GPU.cu) has `e.energy`, `e.theta`, `e.phi`,
  `e.pos`, `e.type` when the electron transmits into `REGION_VAC`.
- CPU: `BackscatterStats` `Datum` carries exit `theta`/`phi`, position, energy and
  the SE1/SE2 type.

The work is: (1) geometry convention, (2) a detector-acceptance abstraction,
(3) accumulators + output, (4) JSON config + CPU/GPU parity.

## 1. Geometry convention

- Beam travels **+z into** the sample; outward optic axis is **-z**. An escaping
  electron has `dir.z < 0`.
- **Take-off polar angle from the optic axis:** `cos(beta) = -dir.z`
  (equivalently `beta = 180 deg - theta`, since `theta = acos(dir.z)`).
  `beta ~ 0` = straight up the column (in-lens); `beta ~ 90 deg` = grazing.
- **Azimuth phi** already available; matters only for side-mounted detectors
  (ETD). Annular / in-lens detectors are azimuthally symmetric.
- Collection depends on the **lab-frame** direction (the detector is fixed in the
  lab), not the local surface normal. Emission physics (local-normal cosine law,
  the protruding-precipitate enhancement) is already handled upstream; the
  detector layer only reads lab-frame `(E, beta, phi)`.

## 2. Detector abstraction

A detector is an acceptance/efficiency function `W_d(E, beta, phi, x, y)` in
`[0, 1]`. Per-pixel signal `S_d = sum over escapes of W_d`. Presets map hardware
to `(E, beta, phi)` windows:

| Preset | Energy window | Angular window | Notes |
| --- | --- | --- | --- |
| `annular_bse` (T1-like) | E > ~50 eV | beta ring from detector geometry | pure geometry accurate for BSE |
| `inlens_se` (T3-like) | E < ~50 eV (opt. band-pass) | small-beta cone | column-funnelled true SE1 |
| `etd_se` (chamber) | E < ~50 eV | azimuthal wedge at moderate beta | needs bias-efficiency proxy |
| `through_lens_bse` | high E | small-beta cone | — |

**Geometric mapping (field-free, straight-line):** a detector that is a disk /
annulus of radius `[r_in, r_out]` at working distance `WD` above the sample
collects an electron iff `r_in <= WD * tan(beta) <= r_out` (and phi within its
coverage). So `{working_distance_mm, inner_radius_mm, outer_radius_mm,
azimuth_range}` reduces to a `(beta, phi)` acceptance band. Exact for BSE.

## 3. Two architectures

**Option A — inline detector filters.** Evaluate each configured `W_d` at escape,
atomic-add the weight into per-detector per-pixel accumulators. Low memory; but
changing a detector requires re-running the Monte Carlo.

**Option B — per-pixel (energy x angle) escape histogram (RECOMMENDED first).**
Accumulate a small `[E-bin x beta-bin (x phi-bin)]` histogram per pixel during the
MC; apply any detector model in fast post-processing (Python) with **zero
re-simulation**. Trades memory (pixels x E-bins x beta-bins) for the ability to
define and iterate detectors after one expensive run. Best fit for detector
hypothesis-testing.

Recommendation: build Option B first (decouples slow MC from detector design),
keep Option A as a later production optimization once detectors are fixed.

## 4. JSON schema sketch (config, not code)

Backward-compatible: absent a `detectors` block, behavior is unchanged (the
current SE/SE1/SE2/BSE outputs are an implicit ideal detector).

```
"detectors": [
  { "name": "T3_inlens_SE", "preset": "inlens_se",
    "energy_min_ev": 0, "energy_max_ev": 50,
    "polar_min_deg": 0, "polar_max_deg": 15 },
  { "name": "annular_BSE", "preset": "annular_bse",
    "energy_min_ev": 50, "energy_max_ev": null,
    "working_distance_mm": 5.0, "inner_radius_mm": 3.0, "outer_radius_mm": 8.0 },
  { "name": "ETD_SE", "preset": "etd_se",
    "energy_max_ev": 50, "polar_min_deg": 30, "polar_max_deg": 70,
    "azimuth_center_deg": 0, "azimuth_width_deg": 60, "bias_efficiency": "low_e_enhanced" }
]
```

Each detector maps to one CSV column and one optional PGM. Option B instead dumps
the per-pixel histogram plus a post-processor that consumes this same list.

## 5. Fidelity ladder and caveats

- **Field-free geometric acceptance is accurate for BSE** (stiff high-energy
  trajectories) and **only approximate for SE** — low-energy SE trajectories are
  dominated by the extraction/immersion field that a static solid-angle window
  does not trace.
- For SE detectors, multiply the geometric window by an **energy/angle
  collection-efficiency curve** as an effective stand-in for the field
  (e.g. the ETD bias that bends low-E SE toward a side detector).
- **SE3 / chamber conversion** (SE created by BSE on the pole-piece/chamber) is a
  separate future module (track BSE to chamber surfaces, generate SE3, collect on
  ETD). It is what makes a chamber ETD read high-Z bright while an in-lens reads
  it dark.
- **SE1/SE2 generation tags** remain as diagnostics; the detector signals become
  the physical observables.

## 6. Code touch-points (for when we build)

- New shared host/device header (`DetectorModel.cuh`, Amphibian pattern): the
  `W_d(E, beta, phi)` evaluator, used identically by CPU and GPU so they agree.
- GPU: extend `KernelConfig` / `GpuOutput` with per-detector accumulators;
  `recordEscape` evaluates detectors (or fills the histogram); device->host copy;
  CSV/PGM writers in CompositeImage.cu.
- CPU: a `DetectorBank` alongside / inside `BackscatterStats` using the same
  evaluator.
- Config parsing in `CompositeImageConfig`; docs in README.md.

## 7. Phased roadmap

1. **Phase 1** — escape-state convention + per-pixel `(E, beta)` histogram
   (Option B) + Python post-processor with the three presets. Lowest risk;
   answers the in-lens-SE vs BSE question on already-simulated data.
2. **Phase 2** — inline `detectors` (Option A), per-detector CSV/PGM, CPU/GPU
   parity.
3. **Phase 3** — energy/angle efficiency curves + ETD bias proxy + azimuthal
   (side-detector) support.
4. **Phase 4 (stretch)** — field-aware SE collection and the SE3/chamber module.

## 8. Validation

- **Limiting case:** a full-acceptance detector (all E, beta 0–90 deg) must
  reproduce today's SE/BSE numbers bit-for-bit.
- **Reproduce observations:** `inlens_se` (low-E, small-beta) -> sigma and W-rich
  M23C6 dark; `annular_bse` -> sigma bright — the bright-BSE / dark-inlens-SE split
  from one run.
- **CPU/GPU agreement** within the existing tolerance; sum of detector coverage
  `<=` total escapes.

## Open decisions before coding

1. Option B (histogram, re-usable) vs Option A (inline) as the first build —
   recommended: B.
2. How far up the fidelity ladder for SE detectors — pure geometry is too crude
   for the ETD; in-lens is acceptable as a first cut.
